3 A C++17 small game library relying on [SFML](https://www.sfml-dev.org) 2.5.x, trying to abstract heavy work (settings up an application, scene management, scripting system, networking...)

5 # Organization of an application
6 
7 An `Application` will handle the window and scenes (through the `SceneManager`).
8 
9 ## Creating a Scene
10 
11 User scenes must inherit from `sgl::Scene`, as in the following example:
12 
13 ~~~~{.cpp}
14 class MyScene : public sgl::Scene
15 {
16 public:
17  // the identifier is a must-have argument, needed to identify a scene
18  MyScene(int id, int test_argument) :
19  sgl::Scene(id),
20  m_argument(test_argument), m_octagon(80.f, 8)
21  {
22  std::cout << "MyScene::MyScene()" << std::endl;
23 
24  m_octagon.setFillColor(sf::Color::Green);
25  m_octagon.setPosition(100, 100);
26  }
27 
28  void onEvent(const sf::Event& event)
29  {
30  if (event.type == sf::Event::KeyPressed)
31  std::cout << "key pressed" << std::endl;
32  }
33 
34  void onUpdate(const sf::Time dt)
35  {}
36 
37  void onRender(sf::RenderTarget& screen)
38  {
39  screen.draw(m_octagon);
40  }
41 
42  void onQuit()
43  {}
44 
45 private:
46  int m_argument;
47  sf::CircleShape m_octagon;
48 };
49 ~~~~
50 
51 The methods `onXXX` can be not implemented, by default they won't do anything.
52 
53 The call order is:
54 * constructor
55 * game loop
56  * event
57  * update
58  * render
59 * quit
60 
61 --------
62 
63 Each scene has a `State` handled by the `SceneManager`, default state is `Stopped` (not rendered, not updated).
64 
65 Only one scene at a time can be in `Running` state, it will be the first one to be rendered and updated, then the `Idle` scenes will be rendered and updated. This is useful to implement a widgets layer for example.
66 Scenes can be manually stopped (if you don't want to run your inventory and your fighting system at the same time, since both will receive events).

68 ## Creating and customizing an Application
69 
70 ~~~~{.cpp}
71 #include <Small/All.hpp>
72 
73 #include <MyScene.hpp>
74 #include <iostream>
75 
76 int main(int argc, char** argv)
77 {
78  const sgl::Settings settings = {};
79  sgl::Application app(settings);
80 
81  app.setTitle("Hello world!")
82  .setVSync(false)
83  .setFPSLimit(60)
84  .setDebug(true);
85 
86  app.add<MyScene>(12);
87  app.setCurrentScene<MyScene>();
88 
89  app.run();
90 
91  return 0;
92 }
93 ~~~~
94 
95 The sgl::Settings can be modified to change the `width` and `height` of the window, as well as its `style` (presence of titlebar, close button...). Some other settings can be modified but are linked to the backend, OpenGL, so modify them only if you know what you are doing.
96 
97 We add and create a new scene to an application by using `app.add<Scene>(args)`, and set it as the current one by using `app.setCurrentScene<MyScene>()`. This means that if we want to use the same scene multiple time with different parameters, for example a `Level` scene, we wouldn't be able to have more than a single level registered at a time since they are selected by the class name and not by their scene id. Thus we can use a workaround with templated classes:
98 
99 ~~~~{.cpp}
100 template <class T, int N>
101 class Wrapper : public T
102 {
103 public:
104  template <typename... Args>
105  Wrapper(Args&&... args) :
106  T(std::forward<Args>(args)...)
107  {}
108 };
109 
110 int main(int argc, char** argv)
111 {
112  const sgl::Settings settings = {};
113  sgl::Application app(settings);
114 
115  app.setTitle("Hello world!")
116  .setVSync(false)
117  .setFPSLimit(60)
118  .setDebug(true);
119 
120  app.add<Wrapper<Level, 1>>(12);
121  app.add<Wrapper<Level, 2>>(15);
122  app.add<Wrapper<Level, 3>>(32);
123  app.setCurrentScene<Wrapper<Level, 1>>();
124 
125  app.run();
126 
127  return 0;
128 }
